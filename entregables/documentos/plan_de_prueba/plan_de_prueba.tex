\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

Este documento provee una vista general del proyecto \textbf{Sistema de
Solicitudes} de la Universidad Autónoma de Zacatecas y describe la estrategia
de pruebas del producto, la lista de los entregables de \emph{testing} y el
plan para llevar a cabo dichas actividades durante el desarrollo.

El Sistema de Solicitudes es una aplicación web monolítica construida con
Django~5 que digitaliza el ciclo de vida de las solicitudes académicas y
administrativas de la institución: catálogo dinámico de tipos de solicitud,
formularios configurables, generación de documentos PDF a partir de plantillas,
máquina de estados con enrutamiento por roles, gestión de archivos adjuntos,
notificaciones por correo y administración de mentores. El presente plan
prescribe el alcance, el enfoque, los recursos y el calendario de todas las
actividades de prueba asociadas, así como los criterios que determinan su
terminación exitosa.

\chapter*{Introducción}
\addcontentsline{toc}{chapter}{Introducción}

El presente Plan de Prueba está diseñado para prescribir el \textbf{alcance}, el
\textbf{enfoque}, los \textbf{recursos} y el \textbf{calendario} de todas las
actividades de \emph{testing} del proyecto Sistema de Solicitudes de la
Universidad Autónoma de Zacatecas.

En el plan se identifican los elementos y características a ser probados, los
tipos de prueba a ejecutar, el personal responsable de las pruebas, los recursos
de hardware y software requeridos, el calendario estimado para completar el
\emph{testing} y los riesgos asociados con su ejecución. El documento sirve como
guía operativa para el equipo y como referencia de cumplimiento frente a la
especificación de requerimientos de software (SRS) y al diseño del sistema
(SDD).

El enfoque de pruebas se apoya en la metodología \textbf{TDD} (Test-Driven
Development): las pruebas se escriben antes que el código de producción, de modo
que cada requerimiento funcional queda respaldado por una o más pruebas
ejecutables. El \emph{stack} de pruebas se compone de \texttt{pytest} con
\texttt{pytest-django} para las pruebas unitarias y de integración,
\texttt{behave} con \texttt{Selenium} para las pruebas funcionales/de aceptación
en navegador, \texttt{JMeter} para las pruebas de performance, \texttt{coverage}
para la medición de cobertura, \texttt{radon} para la complejidad ciclomática y
\texttt{ruff} para la conformidad con PEP\,8.

\subsection{Características que NO serán testeadas}

Las siguientes características quedan \textbf{fuera del alcance} de este plan,
porque corresponden a sistemas de terceros sobre los que el equipo no tiene
control, o porque no forman parte de la especificación de requerimientos de
software del Sistema de Solicitudes. Su exclusión se justifica a continuación.

\begin{itemize}[leftmargin=*]
  \item \textbf{Proveedor de autenticación externo (validación de JWT de
  terceros / SIGA).} El sistema no implementa autenticación propia; recibe y
  confía en un JWT emitido por un proveedor externo (RNF-01) y consulta datos de
  usuario a SIGA (RNF-02). La emisión, firma y validación del token por parte
  del proveedor, así como la disponibilidad de SIGA, son responsabilidad de esos
  servicios y no pueden ser controlados ni reproducidos de forma fiable por el
  equipo. Sí se prueba, en cambio, el \emph{middleware} que decodifica el token
  ya recibido y el \emph{fallback} a los datos del JWT cuando SIGA no responde,
  empleando dobles de prueba.
  \item \textbf{Envío real a través del servidor SMTP externo.} El servidor de
  correo es un servicio de terceros (IE-03). Se prueba el \emph{dispatch} de las
  notificaciones (que el sistema construye y entrega el mensaje al backend de
  correo) y el comportamiento de \emph{fallback} ante una falla de SMTP (la
  operación de negocio no debe verse afectada), pero no se prueba la entrega
  efectiva del correo por parte del servidor de un tercero.
  \item \textbf{Interfaces de hardware.} El sistema es una aplicación web que no
  interactúa directamente con dispositivos de hardware; no existen controladores
  ni interfaces físicas que probar.
\end{itemize}

\section{Tipo de Test}

En el proyecto Sistema de Solicitudes se ejecutan \textbf{cuatro} tipos de
prueba a lo largo del desarrollo:

\begin{itemize}[leftmargin=*]
  \item \textbf{(a) Pruebas Unitarias.} Verifican de forma aislada el
  comportamiento de las unidades de software más pequeñas (servicios,
  repositorios, formularios, esquemas Pydantic, funciones de dominio). Se
  desarrollan de manera colaborativa: cada integrante escribe las pruebas de los
  módulos a su cargo. Se implementan con \texttt{pytest} + \texttt{pytest-django}
  (sobre el estilo \texttt{TestCase} cuando aplica) y constituyen el grueso de la
  suite, con \textbf{769 pruebas} (\textbf{714 funciones de test únicas}).

  \item \textbf{(b) Pruebas Funcionales / de Aceptación.} Simulan el
  comportamiento del sistema tal como lo experimenta el usuario final, para
  evaluar la conformidad con los requerimientos especificados y las historias de
  usuario. Se redactan en \textbf{Gherkin} y se ejecutan con \textbf{behave} y
  \textbf{Selenium}, controlando un navegador real. La suite de aceptación cubre
  \textbf{7 \emph{features}} y \textbf{20 escenarios} que recorren los flujos
  principales de los cinco roles (creación, consulta, atención, cambio de estado
  y administración).

  \item \textbf{(c) Pruebas de Performance.} Evalúan el comportamiento del
  sistema bajo carga y concurrencia. Se ejecutan con \textbf{JMeter} sobre
  \textbf{2 casos}: (1) \emph{login} con 10 usuarios concurrentes y (2) alta de
  solicitud con 5 usuarios concurrentes. Se observan tiempos de respuesta, tasa
  de error y \emph{throughput}.

  \item \textbf{(d) Pruebas de Integración.} Verifican que los distintos
  componentes colaboren correctamente entre sí. En la arquitectura por capas de
  Django (View~$\rightarrow$~Service~$\rightarrow$~Repository sobre el ORM), las
  pruebas de integración están \textbf{entremezcladas con las unitarias}: al
  probar una vista a través del cliente de pruebas de Django se integran de
  hecho el servicio, el repositorio y el ORM contra una base de datos real, así
  como la serialización de DTOs Pydantic y el \emph{rendering} de plantillas.
  Estas pruebas se identifican con el \emph{marker} \texttt{integration} de
  \texttt{pytest}, lo que permite ejecutarlas o aislarlas de forma selectiva
  (\texttt{pytest -m integration}).
\end{itemize}

\section{Riesgos y Problemas}

\begin{center}
\renewcommand{\arraystretch}{1.4}
\begin{tabularx}{\textwidth}{|L{5.6cm}|X|}
\hline
\rowcolor{uazblue!12}
\textbf{Riesgo} & \textbf{Mitigación} \\
\hline
Dependencia del proveedor de autenticación externo (JWT) y de SIGA: una falla o
indisponibilidad bloquearía las pruebas que requieren un usuario autenticado. &
Aislar la dependencia mediante dobles de prueba (tokens fabricados, respuestas
de SIGA simuladas con \texttt{responses}); probar el \emph{fallback} a datos del
JWT y no depender de los servicios reales en la suite automatizada. \\
\hline
Fallas del servidor SMTP externo durante las pruebas de notificación. & Usar el
\emph{backend} de correo en memoria de Django en pruebas; verificar únicamente
el \emph{dispatch} y el comportamiento de \emph{fallback}, garantizando que la
operación de negocio no se vea afectada. \\
\hline
Datos sembrados (\emph{seed}) inconsistentes o insuficientes que invaliden los
escenarios. & Construir datos con \texttt{model\_bakery} y \emph{fixtures}
deterministas; congelar el tiempo con \texttt{freezegun} donde el folio o las
fechas sean relevantes. \\
\hline
Tiempo limitado del equipo frente al volumen de casos a cubrir. & Priorizar por
criticidad (flujos núcleo CU-01, CU-07, CU-08 primero); paralelizar el trabajo
por módulo entre los integrantes; aprovechar TDD para evitar retrabajo. \\
\hline
Pruebas \emph{flaky} (intermitentes) en navegador con Selenium por
sincronización o tiempos de carga. & Emplear esperas explícitas, fijar la
versión de Chrome/\emph{driver}, ejecutar en un entorno Docker reproducible y
reintentar escenarios marcados como inestables antes de declararlos fallidos. \\
\hline
\end{tabularx}
\end{center}

\section{Logística de Pruebas}

\subsection{¿Quiénes realizarán las pruebas?}

Para este proyecto las pruebas serán realizadas por los propios miembros del
equipo de desarrollo, quienes fungen con el rol de \emph{testers} con el
objetivo de optimizar los costos y los tiempos del proyecto. Cada integrante es
responsable de escribir y mantener las pruebas del módulo a su cargo, además de
participar en la construcción de la suite de aceptación compartida.

\subsection{¿Cuándo ocurrirán las pruebas?}

Conforme a la metodología \textbf{TDD}, las pruebas se escriben \textbf{antes}
que el código de producción y se ejecutan de forma continua durante todo el
desarrollo. Los \emph{testers} del equipo pueden comenzar la ejecución de las
pruebas cuando las siguientes entradas se encuentren listas:

\begin{itemize}[leftmargin=*]
  \item Se cuenta con el ambiente de pruebas de Django configurado y
  reproducible (contenedores Docker con la base de datos de pruebas).
  \item De acuerdo con TDD, el desarrollo del software se realiza a la par de la
  escritura de las pruebas (\emph{prueba primero}: RED\,$\rightarrow$\,GREEN\,$\rightarrow$\,REFACTOR).
  \item Se cuenta con la definición de los requerimientos técnicos y los
  criterios de aceptación de las historias de usuario del proyecto.
  \item Se dispone de los datos de prueba y \emph{fixtures} necesarios para los
  escenarios.
\end{itemize}

\chapter{Objetivos del Test}

Los objetivos de la prueba son verificar la funcionalidad del Sistema de
Solicitudes de la UAZ y, mediante la implementación de la metodología TDD,
desarrollar dicha funcionalidad a partir de la creación de diferentes pruebas:
pruebas unitarias, pruebas de integración, pruebas funcionales/de aceptación y
pruebas de performance.

El proyecto se enfoca en probar la funcionalidad descrita en las historias de
usuario y casos de uso del SRS, teniendo en cuenta el manejo de las operaciones
principales del sistema:

\begin{itemize}[leftmargin=*]
  \item \textbf{Catálogo y formularios:} gestión de tipos de solicitud,
  formularios dinámicos y plantillas (CU-10, CU-11, CU-12).
  \item \textbf{Ciclo de vida de solicitudes:} creación con folio único,
  consulta, detalle, cancelación y seguimiento (CU-01 a CU-05).
  \item \textbf{Atención por el personal:} listado de asignadas, atención,
  cambio de estado y generación de PDF, con enrutamiento por rol (CU-06 a
  CU-09).
  \item \textbf{Administración:} mentores, dashboard de métricas y exportación
  de reportes (CU-13, CU-14, CU-15).
  \item \textbf{Notificaciones:} despacho de correos por evento y
  \emph{fallback} ante fallas (CU-17).
\end{itemize}

El propósito es garantizar que estas operaciones se implementen correctamente y
funcionen de manera estable en un entorno real de trabajo, respetando las reglas
de negocio y la separación de responsabilidades por rol.

\chapter{Criterio de Prueba}

\section{Criterio de Suspensión}

En caso de que los miembros del equipo detecten que las pruebas no están
contribuyendo positivamente al desarrollo del proyecto conforme a la metodología
TDD —por ejemplo, ante una tasa elevada de fallos no atribuibles al código bajo
prueba, dependencias externas inestables o un ambiente de pruebas roto— se
suspenderá la ejecución, se revisarán los apartados descritos por TDD, se harán
las correcciones necesarias en las pruebas o en el ambiente y se retomará el
ciclo RED\,$\rightarrow$\,GREEN\,$\rightarrow$\,REFACTOR para continuar con el
desarrollo del proyecto.

\section{Criterio de Salida}

Definen los criterios que denotan una terminación exitosa del desarrollo del
proyecto con sus respectivas pruebas:

\begin{itemize}[leftmargin=*]
  \item De acuerdo con la metodología TDD, las pruebas definidas cubren el
  \textbf{100\,\%} de las funcionalidades requeridas del proyecto, con un
  objetivo de \textbf{cobertura de 100\,\%} medida con \texttt{coverage} sobre
  modelos, vistas, formularios y servicios (reporte de cobertura adjunto a los
  entregables).
  \item Las pruebas de aceptación con \textbf{behave} y \textbf{Selenium} cubren
  las historias de usuario descritas en el SRS; las pruebas unitarias se
  implementan con \texttt{TestCase}/\texttt{pytest}.
  \item Todos los métodos del sistema presentan una \textbf{complejidad
  ciclomática $\leq$ 10}, verificada con \texttt{radon}.
  \item El código cumple con el estándar de estilo \textbf{PEP\,8}, verificado
  con \texttt{ruff}.
  \item La totalidad de la suite automatizada (unitarias, integración y
  aceptación) pasa en verde en el ambiente de pruebas, y las pruebas de
  performance con JMeter se ejecutan dentro de umbrales aceptables de tiempo de
  respuesta y tasa de error.
\end{itemize}

\chapter{Ambiente de Pruebas}

Cada integrante del equipo cuenta con un equipo de cómputo con las
características mínimas requeridas y con \textbf{Docker} instalado, lo que
permite levantar de forma idéntica el ambiente de la aplicación (servicio
\emph{web} de Django, base de datos PostgreSQL y MailHog para inspección de
correo) y ejecutar la suite de pruebas siguiendo la metodología TDD. El uso de
contenedores garantiza que las pruebas se ejecuten sobre un entorno
reproducible, eliminando las diferencias entre las máquinas de los
desarrolladores.

Las pruebas unitarias, de integración y de aceptación se ejecutan de manera
\textbf{local} dentro de los contenedores Docker de cada integrante. Para las
pruebas de performance con JMeter se contempla, además, la disposición de un
\textbf{servidor de pruebas / preproducción} que reproduzca condiciones más
cercanas a producción, de modo que las mediciones de carga y concurrencia sean
representativas y no se vean distorsionadas por los recursos de una máquina de
desarrollo.

artefactos:

  $\leftrightarrow$ pruebas.

  \item \textbf{Reportes de cobertura} (\texttt{coverage}, objetivo 100\,\%) y de
  \textbf{complejidad ciclomática} (\texttt{radon}, métodos $\leq$ 10), junto con
  la verificación de estilo PEP\,8 (\texttt{ruff}).
  \item Documentación de soporte: \textbf{SRS}, \textbf{SDD}, documento de
  \textbf{Visión y Alcance}, \textbf{esquema de la base de datos} y
