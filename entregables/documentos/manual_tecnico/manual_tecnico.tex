\chapter{Introducción y propósito}
% ============================================================================

El \textbf{Sistema de Solicitudes} de la Universidad Autónoma de Zacatecas (UAZ)
es una aplicación web monolítica construida con Django que permite a alumnos y
docentes presentar solicitudes académicas y administrativas mediante formularios
dinámicos, dar seguimiento a su ciclo de vida con una máquina de estados, generar
documentos PDF a partir de plantillas y gestionar todo el flujo con base en roles.

Este \textbf{Manual Técnico} está dirigido al personal de desarrollo,
mantenimiento y operación del sistema. Su propósito es documentar, de forma
verificable y reproducible:

\begin{itemize}[leftmargin=1.4em]
  \item Los requisitos del entorno de ejecución y las versiones del stack.
  \item La arquitectura técnica y la organización del código.
  \item El procedimiento de instalación y despliegue con Docker.
  \item Los comandos de operación diaria.
  \item Las variables de configuración.
  \item La estrategia de pruebas (unitarias, integración, aceptación,
        rendimiento) y de aseguramiento de calidad.
  \item Las tareas de mantenimiento y extensión.
  \item La solución de los problemas más comunes.
\end{itemize}

\paragraph{Convenciones.} Los identificadores de código (variables, funciones,
clases, módulos) están en inglés; la copia visible para el usuario final está en
español. A lo largo del manual, todo comando mostrado es ejecutable tal cual
contra el repositorio; cada atajo de \texttt{make} se acompaña de su equivalente
crudo de \texttt{docker compose}, de modo que no es indispensable tener
\texttt{make} instalado.

\chapter{Arquitectura técnica}
% ============================================================================

\section{Estilo arquitectónico}

El sistema sigue una arquitectura por capas \textbf{Vista $\rightarrow$ Servicio
$\rightarrow$ Repositorio} (\emph{View $\rightarrow$ Service $\rightarrow$
Repository}), con DTOs de Pydantic v2 en cada frontera entre capas:

\begin{itemize}[leftmargin=1.4em]
  \item \textbf{Vista (View).} Recibe la petición HTTP, valida la entrada con
        formularios de Django y delega la lógica al servicio. Renderiza plantillas
        del lado del servidor a partir de DTOs de Pydantic (no hay DRF ni API JSON
        por defecto).
  \item \textbf{Servicio (Service).} Contiene la lógica de negocio y las reglas de
        transición de estado. No conoce HTTP ni el ORM; opera sobre DTOs.
  \item \textbf{Repositorio (Repository).} Único lugar donde vive el ORM de
        Django. Traduce entre modelos del ORM y DTOs.
\end{itemize}

Todas las excepciones heredan de \texttt{\_shared.exceptions.AppError} y un
\emph{middleware} las mapea a respuestas HTTP. La validación de JWT se realiza
también en \emph{middleware} (el proveedor de identidad es externo).

\section{Stack tecnológico}

\begin{itemize}[leftmargin=1.4em]
  \item \textbf{Python 3.13 + Django 5.x} con plantillas del lado del servidor
        (sin DRF).
  \item \textbf{Pydantic v2} para los DTOs en cada frontera de capa.
  \item \textbf{PostgreSQL} en producción y desarrollo; \textbf{SQLite} para la
        suite de pruebas en proceso. El ORM queda contenido en los repositorios.
  \item \textbf{Tailwind CSS v4} + \textbf{Alpine.js v3} + iconos \textbf{Lucide}
        para la interfaz. El binario standalone de Tailwind se ejecuta dentro del
        contenedor \texttt{web} (no requiere Node).
  \item \textbf{WeasyPrint} para la generación de PDF a partir de plantillas
        HTML/CSS.
  \item \textbf{Validación de JWT} en \emph{middleware} (proveedor de autenticación
        externo).
  \item \textbf{Celery + Redis} para correo asíncrono (opcional).
  \item \textbf{pytest + pytest-django}, con \texttt{model\_bakery},
        \texttt{freezegun} y \texttt{responses} para las pruebas.
\end{itemize}

\section{Aplicaciones Django}

El proyecto se organiza en una aplicación Django por dominio:

\begin{longtable}{L{3.2cm} L{11cm}}
\toprule
\textbf{App} & \textbf{Responsabilidad} \\
\midrule
\endhead
\texttt{\_shared} & Infraestructura transversal: excepciones (\texttt{AppError}),

\texttt{usuarios} & Validación de JWT, roles, perfil y directorio de usuarios.

\chapter{Instalación y despliegue}
% ============================================================================

El stack de desarrollo se levanta por completo en Docker: \texttt{nginx}
(terminación TLS en el puerto \texttt{443}), \texttt{web} (Django), \texttt{db}
(PostgreSQL) y \texttt{mailhog} (captura de SMTP en el puerto \texttt{8025}).

\section{Requisitos previos}

\begin{itemize}[leftmargin=1.4em]
  \item Docker Desktop (o Docker Engine 24+ con Compose v2).
  \item \texttt{git}.
\end{itemize}

Los certificados TLS de desarrollo para \texttt{localhost} ya están incluidos en
el repositorio; no es necesario generarlos.

\section{Instalación paso a paso}

\subsection{1. Levantar el stack}

\begin{lstlisting}[language=bash]
docker compose -f docker-compose.dev.yml up -d --build
\end{lstlisting}

\subsection{2. Aplicar migraciones}

\begin{lstlisting}[language=bash]
docker compose -f docker-compose.dev.yml exec -T web python manage.py migrate
\end{lstlisting}

\subsection{3. Sembrar datos de desarrollo}

\begin{lstlisting}[language=bash]
docker compose -f docker-compose.dev.yml exec -T web python manage.py seed
\end{lstlisting}

El sembrado es \textbf{idempotente}: volver a ejecutarlo preserva las filas
agregadas a mano. Para reconstruir desde cero, use \texttt{--fresh}:

\begin{lstlisting}[language=bash]
docker compose -f docker-compose.dev.yml exec -T web python manage.py seed --fresh
\end{lstlisting}

Para ejecutar el sembrador de una sola app:

\begin{lstlisting}[language=bash]
docker compose -f docker-compose.dev.yml exec -T web python manage.py seed --only usuarios
\end{lstlisting}

\subsection{4. Acceder al sistema}

Abra \url{https://localhost/auth/dev-login} y haga clic en cualquier rol para
iniciar sesión. El navegador puede advertir sobre el certificado autofirmado;
acéptelo una vez.

\section{Servicios del stack}

\paragraph{Nginx con TLS.} El servicio \texttt{nginx-dev} termina TLS en
\texttt{:443} (y expone \texttt{:80}), usando los certificados autofirmados de
\texttt{certs/} montados en modo solo lectura. Hace de \emph{proxy} hacia el
contenedor \texttt{web} (Django en \texttt{:8000}) y hacia \texttt{mailhog}.

\paragraph{Mailhog.} Captura todo el correo saliente en desarrollo. Su bandeja se
consulta en \url{http://localhost:8025/}; es útil para inspeccionar las
notificaciones que dispara el sistema al crear o transicionar solicitudes.

\paragraph{Base de datos.} El servicio \texttt{db} corre PostgreSQL con un
\emph{healthcheck} (\texttt{pg\_isready}); el contenedor \texttt{web} espera a que
esté sano antes de arrancar. Los datos persisten en el volumen
\texttt{solicitudes\_dev\_data}.

\paragraph{Web.} El contenedor \texttt{web} arranca el binario standalone de
Tailwind en modo \emph{watch} y el servidor de desarrollo de Django de forma
simultánea, montando \texttt{./app} como volumen para recarga en caliente.

\chapter{Pruebas}
% ============================================================================

El proyecto cuenta con varios niveles de prueba, cada uno con su propio comando.

\section{Pruebas unitarias y de integración}

Se ejecutan con \texttt{pytest} dentro del contenedor \texttt{web} (en proceso,
sobre SQLite):

\begin{lstlisting}[language=bash]
make test
# equivalente:
docker compose -f docker-compose.dev.yml exec -T web pytest
\end{lstlisting}

Las pruebas se etiquetan con \emph{markers}. Para excluir las de navegador se usa
\texttt{-m "not e2e"}; para ejecutar solo las de navegador, \texttt{-m e2e}. La
suite de extremo a extremo en proceso usa Playwright y requiere un arranque único
de Chromium dentro del contenedor:

\begin{lstlisting}[language=bash]
make e2e-install     # instala Chromium + dependencias (una vez)
make e2e             # corre la suite Playwright (Tier 1 + Tier 2)
make e2e-headed      # con navegador visible (--headed --slowmo 200)
\end{lstlisting}

También puede ejecutarse contra un PostgreSQL efímero con
\texttt{make e2e-postgres}.

\section{Cobertura}

\begin{lstlisting}[language=bash]
make coverage         # reporte en terminal con lineas faltantes
make coverage-html    # reporte HTML en app/htmlcov/
\end{lstlisting}

Pasando \texttt{E2E=1} (por ejemplo \texttt{make coverage E2E=1}) se añade la suite
Playwright a la medición. La cobertura global medida del proyecto es del
\textbf{94\%} sobre 5515 sentencias.

\section{Pruebas de aceptación (behave + Selenium)}

Las pruebas de aceptación de extremo a extremo están escritas en \textbf{Gherkin
(español)} con \textbf{behave} y se ejecutan con \textbf{Selenium} sobre Google
Chrome contra el stack en ejecución. Viven en la carpeta \texttt{acceptance/} y
cubren una historia por módulo del equipo:

\begin{longtable}{L{4cm} L{4cm} L{6cm}}
\toprule
\textbf{Feature} & \textbf{Módulo} & \textbf{Historia} \\
\midrule
\endhead
\texttt{acceso.feature} & usuarios & Inicio de sesión por rol / bloqueo de anónimos. \\
\texttt{intake.feature} & solicitudes/intake & Alta de solicitudes desde el alumno. \\
\texttt{archivos.feature} & solicitudes/archivos & Adjuntar y descargar archivos (con autorización). \\
\texttt{revision.feature} & solicitudes/revision & Cola de revisión y atención del personal. \\
\texttt{plantillas.feature} & solicitudes/pdf & Catálogo y editor de plantillas PDF. \\
\texttt{mentores.feature} & mentores & Importación CSV de mentores. \\
\texttt{reportes.feature} & reportes & Dashboard administrativo y export CSV. \\
\bottomrule
\end{longtable}

\paragraph{Requisitos.} El stack debe estar levantado y sembrado
(\texttt{make up \&\& make seed}); la suite ataca \texttt{https://localhost} (el
\emph{driver} ignora el certificado autofirmado con \texttt{acceptInsecureCerts}).
Google Chrome debe estar instalado (Selenium Manager descarga el \emph{driver}).
Las dependencias se instalan en un \emph{virtualenv} aparte del proyecto Django:

\begin{lstlisting}[language=bash]
cd acceptance
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
\end{lstlisting}

\paragraph{Ejecución.}

\begin{lstlisting}[language=bash]
cd acceptance
.venv/bin/behave                          # toda la suite
.venv/bin/behave features/intake.feature  # una sola feature
.venv/bin/behave -D headless=false        # ver el navegador
.venv/bin/behave -D base_url=https://localhost   # apuntar a otro entorno
\end{lstlisting}

Las aserciones se hacen siempre sobre lo que \textbf{ve el usuario} en el
navegador (textos, URLs, páginas de error), nunca sobre códigos HTTP crudos.

\section{Pruebas de rendimiento (JMeter)}

Las pruebas de carga se ejecutan con Apache JMeter a partir del plan de prueba
\texttt{solicitudes\_performance.jmx}, que ejercita los flujos críticos (login,
alta de solicitud, cola de revisión, descarga de PDF) contra el stack en
ejecución. Para correrlas sin interfaz gráfica:

\begin{lstlisting}[language=bash]
jmeter -n -t solicitudes_performance.jmx -l resultados.jtl -e -o reporte_html
\end{lstlisting}

El reporte HTML resultante (\texttt{reporte\_html/}) muestra latencias,
\emph{throughput} y porcentaje de error por petición.

\section{Complejidad ciclomática y mantenibilidad (radon)}

La complejidad ciclomática y el índice de mantenibilidad se miden con
\textbf{radon}:

\begin{lstlisting}[language=bash]
radon cc app/ -s -a      # complejidad ciclomatica (con promedio)
radon mi app/            # indice de mantenibilidad
\end{lstlisting}

El código se mantiene en rango \textbf{A} de mantenibilidad en prácticamente todos
los módulos, y la complejidad ciclomática por función es baja (rango A en la gran
mayoría).

\section{Estilo (PEP 8) y tipado}

El cumplimiento de estilo se verifica con \textbf{ruff} y el tipado estático con
\textbf{mypy} en modo estricto:

\begin{lstlisting}[language=bash]
make lint    # ruff check .
make type    # mypy .
\end{lstlisting}

\section{Añadir una nueva característica respetando las capas}

Toda característica nueva debe respetar el flujo \textbf{Vista $\rightarrow$
Servicio $\rightarrow$ Repositorio} y usar DTOs de Pydantic en las fronteras. El
procedimiento recomendado es:

\begin{enumerate}[leftmargin=1.5em]
  \item \textbf{Modelo y migración.} Defina o modifique el modelo del ORM en
        \texttt{models/} y genere la migración con \texttt{make makemigrations},
        luego \texttt{make migrate}.
  \item \textbf{DTOs.} Declare los esquemas de Pydantic en \texttt{schemas.py}
        para la entrada y la salida del servicio.
  \item \textbf{Repositorio.} Implemente el acceso a datos en
        \texttt{repositories/}, manteniendo el ORM contenido aquí; traduzca entre
        modelos y DTOs.
  \item \textbf{Servicio.} Codifique la lógica de negocio en \texttt{services/},
        operando solo sobre DTOs (sin HTTP ni ORM).
  \item \textbf{Excepciones.} Defina las excepciones de dominio en
        \texttt{exceptions.py} como subclases de \texttt{AppError}, con su
        \texttt{user\_message} en español.
  \item \textbf{Vista y plantilla.} Implemente la vista en \texttt{views/},
        validando la entrada con un formulario de Django y renderizando la
        plantilla a partir de los DTOs.
  \item \textbf{Cableado.} Registre las dependencias en \texttt{dependencies.py} y
        las rutas en \texttt{urls.py}.
  \item \textbf{Pruebas.} Agregue pruebas en \texttt{tests/} y verifique con
        \texttt{make test} y \texttt{make coverage}.
\end{enumerate}

\paragraph{Añadir una app completa.} Cree la carpeta bajo \texttt{app/} replicando
la estructura por capas, regístrela en \texttt{INSTALLED\_APPS}
(\texttt{config/settings/base.py}) e incluya sus \texttt{urls.py} en
\texttt{config/urls.py}.

\section{Datos sembrados (seeders)}

Para sembrar datos de una app nueva, agregue un \texttt{seeders.py} en la raíz de
la app con una función \texttt{run(*, fresh: bool) -> None}. El comando
\texttt{manage.py seed} lo descubre automáticamente:

\begin{lstlisting}[language=python]
# app/<su_app>/seeders.py
DEPENDS_ON: list[str] = ["usuarios"]  # opcional: corre despues de estas apps

def run(*, fresh: bool) -> None:
    if fresh:
        ...  # borra las filas que este seeder posee
    ...      # update_or_create de sus filas de muestra
\end{lstlisting}

Use \texttt{update\_or\_create} para mantener la idempotencia y borre filas solo
cuando \texttt{fresh} es verdadero.

\section{Migraciones}

Genere migraciones tras cualquier cambio de modelo con \texttt{make makemigrations}
y aplíquelas con \texttt{make migrate}. Las migraciones se versionan junto con el
código en la carpeta \texttt{migrations/} de cada app.

\section{Advertencia de certificado autofirmado}

El \emph{stack} de desarrollo usa nginx con un certificado autofirmado para
\texttt{localhost}. El navegador advertirá que la conexión no es de confianza la
primera vez; acepte la excepción una sola vez. Para las pruebas de aceptación, el
\emph{driver} de Selenium ya ignora el certificado mediante
\texttt{acceptInsecureCerts}.

\section{Puertos ocupados}

El \emph{stack} expone los puertos \texttt{80}, \texttt{443}, \texttt{5432} y
\texttt{8025} en el anfitrión. Si alguno está ocupado, el arranque fallará. Detenga
el proceso que lo usa o ajuste el mapeo de puertos en
\texttt{docker-compose.dev.yml}. Para verificar qué proceso ocupa un puerto:

\begin{lstlisting}[language=bash]
lsof -i :443
\end{lstlisting}

\section{Reconstruir la imagen}

Si cambian las dependencias (\texttt{requirements.txt} /
\texttt{requirements-dev.txt}) o el \texttt{Dockerfile}, reconstruya la imagen:

\begin{lstlisting}[language=bash]
make build
# o forzando una reconstruccion completa:
docker compose -f docker-compose.dev.yml build --no-cache web
\end{lstlisting}

\section{Estado inconsistente o base corrupta en desarrollo}

Para partir de cero (detiene todo y borra los volúmenes, incluida la base de
datos) y volver a levantar:

\begin{lstlisting}[language=bash]
make clean
make up
make migrate
make seed
\end{lstlisting}

\section{Chromium ausente en pruebas de navegador}

Si las pruebas Playwright fallan por falta de navegador, ejecute el arranque único:

\begin{lstlisting}[language=bash]
make e2e-install
\end{lstlisting}
